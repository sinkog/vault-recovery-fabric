vault-recovery-fabric deployed to namespace {{ .Release.Namespace }}.

IMPORTANT: This chart requires the Helm release name to be "vault".
The HashiCorp Vault subchart generates service names from the release name:
  vault-active             (active node service)
  vault-0.vault-internal   (headless internal service)

These names are hardcoded in the init, recovery, and rekey job scripts.
Using a different release name will break the recovery flow.

  Supported: helm upgrade -i vault ./vault -n {{ .Release.Namespace }}
  Not supported: helm upgrade -i vault-recovery-fabric ./vault -n {{ .Release.Namespace }}

=== INIT MATERIAL RETRIEVAL ({{ .Values.bootstrap.holdInitMaterialSeconds }}s window) ===
On fresh install, the bootstrap job holds /tmp for {{ .Values.bootstrap.holdInitMaterialSeconds }}s.
Retrieve with kubectl cp while vault-wait-job pod is running:

  JOB=$(kubectl get pod -n {{ .Release.Namespace }} -l app=vault-wait \
    -o jsonpath='{.items[0].metadata.name}')
  kubectl cp {{ .Release.Namespace }}/$JOB:/tmp/vault-token.txt  ./vault-root-token.txt
  kubectl cp {{ .Release.Namespace }}/$JOB:/tmp/vault-unseal.txt ./vault-unseal-keys.txt

Store these securely. The files are deleted when the hold period expires.

NOTE: helm install/upgrade requires --timeout to match holdInitMaterialSeconds:
  helm upgrade -i vault ./vault -n {{ .Release.Namespace }} --timeout {{ .Values.bootstrap.holdInitMaterialSeconds }}s
{{- end }}

UPGRADE NOTE: If upgrading from a version that predates the vault-bootstrap role,
the bootstrap job post-config phase will fail with "vault-bootstrap auth failed".
One-time fix: obtain a root or admin token and run manually:
  vault policy write vault-bootstrap /path/to/vault-bootstrap.hcl
  vault write auth/kubernetes/role/vault-bootstrap \
    bound_service_account_names=vault-recovery \
    bound_service_account_namespaces={{ .Release.Namespace }} \
    policies=vault-bootstrap ttl=10m token_num_uses=5
Then re-run: helm upgrade vault ./vault -n {{ .Release.Namespace }} --reuse-values
